# 参考文献核查工具 (Academic Cite Checker)

## 1. 背景与问题

### 什么是"文献幻觉"？

大语言模型在生成学术内容时，经常会**编造看似真实但实际上完全不存在的论文引用**。这些幻觉文献通常具有以下特征：
- 作者名字看起来合理（如知名学者）
- 标题听起来专业且符合上下文
- 期刊/会议名称真实存在
- 年份和卷期号格式正确
- **但 DOI 无法解析，或在任何数据库中都找不到**

### 为什么这很危险？

对于学生来说，在毕业论文或学术论文中引用幻觉文献可能导致：
- **学术诚信问题**：被认定为抄袭或数据造假
- **论文被拒**：审稿人核查引用时发现不存在
- **学术声誉受损**：影响未来的学术和职业发展

## 2. 核心设计理念

### 为什么采用"片段判定法"？

传统的方法会打开每个搜索结果页面逐一查看详情，这会产生两个问题：
1. **时间开销大**：每个网页加载可能需要 2-5 秒，对于 50 条引用就需要几分钟
2. **不必要的精确**：实际上，只要标题、年份和第一作者匹配，就能 99% 确定是同一条文献

**片段判定法的优势**：
- **速度**：搜索 API 返回的片段（snippet）已经包含关键信息
- **准确**：标题+年份+第一作者的组合具有极高的唯一性
- **可扩展**：可以轻松并行处理数十条引用

### 为什么使用"工具并行化"？

假设一个 .bib 文件有 30 条引用，串行处理每条引用（解析→搜索→验证→判定）可能需要 30×10=300 秒（5 分钟）。而并行处理可以：
- **第一轮**：一次性生成所有 30 条工具调用
- **第二轮**：并行执行所有搜索和验证
- **总时间**：取决于最慢的那条，通常只需 10-20 秒

## 3. SOP (Ultra-Fast Workflow)

### 第一回合：全量分流 (One-Shot Dispatch)

**核心原则**：在 1 个 Turn 内完成所有工具调用的生成，禁止逐步思考。

**动作**：
1. **调用 DOI 验证脚本**：
   ```
   python scripts/verify_doi.py '<JSON 数组>'
   ```
   注意：输入是一个 JSON 数组字符串，包含所有带 DOI 的条目。

2. **并行搜索无 DOI 条目**：
   对于每条没有 DOI 的引用，同时生成 `search(source="scholar")` 调用。
   搜索查询格式：`"标题" 第一作者 年份`

**禁止**：
- 严禁在搜索前描述步骤（如"现在我将对第一条引用进行搜索..."）
- 严禁一条条串行搜索
- 严禁请求用户确认后再继续

### 第二回合：片段判案 (Snippet-Based Verdict)

**判定策略**：

1. **DOI 验证结果（最高优先级）**：
   - 直接采信 `verify_doi.py` 脚本的结论
   - 如果脚本返回 "DOI not found" → 标记为 ❌ Fake/Not Found
   - 如果脚本返回字段不匹配 → 标记为 ⚠️ Mismatched

2. **搜索结果判定（基于片段）**：
   - 对比 `search` 返回的 Snippet 片段
   - **匹配标准**：标题（允许少量词汇差异）+ 年份 + 第一作者
   - 如果三个字段都匹配 → 标记为 ✅ True
   - 如果有任一字段不匹配 → 标记为 ⚠️ Mismatched
   - 如果搜索结果为空 → 标记为 ❌ Fake/Not Found

**判官决策矩阵 (Pre-defined Logic)**：

| 场景 | 判定 | 说明 |
|------|------|------|
| 标题+年份+第一作者完全匹配 | ✅ | 文献真实存在 |
| 标题匹配但年份/作者不同 | ⚠️ | 可能张冠李戴，需人工检查 |
| 搜索结果包含 Workshop/CVPRW 而用户未写 | ✅ | 刊名微差，建议优化但判定为真 |
| 用户提供 arXiv，搜索结果是正式会议 | ⚠️ | 提供正式版 BibTeX 建议 |
| 无任何搜索结果 | ❌ | 极可能是幻觉文献 |

**禁令**：
- **严禁使用 `agent-browser` 访问网页**验证搜索结果
- 除非 `search` 接口返回零结果且必须获取 BibTeX 时，才允许访问具体页面

### 第三阶段：结果静默输出 (Silent Report)

**进度输出**：
- 执行中：每批次仅输出一行进度 `[Progress] 正在处理 [X/Total]...`
- 严禁输出任何其他描述性文字

**最终交付**：
- 所有比对证据、更正后的 BibTeX 和链接必须写入 `verification_report.md`
- 对话框仅输出统计摘要（三行）+ 报告文件链接

## 4. 判定标准

| 状态 | 含义 | 处理建议 |
|------|------|----------|
| ✅ **True** | 核心字段一致（允许 Workshops 等刊名微差） | 引用可信，无需修改 |
| ⚠️ **Mismatched** | 文献存在但有实质错误（年份/作者/刊名张冠李戴） | 需人工核查，根据官方数据修正 |
| ❌ **Fake/Not Found** | 权威数据库无此记录 | **极可能是幻觉文献，必须删除或替换** |
| ❓ **Unknown** | 无法自动判定（如搜索服务不可用） | 建议人工逐一核查 |

## 5. 性能约束 (Speed Constraints)

**严禁过度推理**：
- Agent 禁止解释比对过程（如"我发现第一条引用的标题是...而搜索结果是...因此我判断..."）
- 所有判定逻辑必须内化为预定义规则，直接输出结论

**脚本优先**：
- 脚本结果为最高优先级，严禁在脚本结果基础上进行二次推理验证

**搜索片段化**：
- 只要 Snippet 能证明文献存在，禁止点开网页链接获取"更多细节"

## 6. 参考知识库 (Reference Knowledge Base)

Agent 在核查时应优先识别并参考以下官方源：

| 数据库 | 适用场景 | URL 特征 |
|--------|----------|----------|
| **CVF (Computer Vision Foundation)** | CVPR, ICCV, WACV 等视觉顶会 | `openaccess.thecvf.com` |
| **IEEE Xplore** | 电气电子、计算机工程期刊 | `ieeexplore.ieee.org` |
| **ACM Digital Library** | 计算机科学综合 | `dl.acm.org` |
| **arXiv** | 预印本（需核实是否已有正式发表版） | `arxiv.org` |
| **Crossref** | 通用 DOI 解析 | `api.crossref.org` |

**识别技巧**：
- 如果文献标题包含 "CVPR", "ICCV", "ECCV", "WACV" → 优先查 CVF
- 如果文献标题包含 "IEEE" → 优先查 IEEE Xplore
- 如果文献标题包含 "ACM", "SIGGRAPH", "SIGIR", "CHI" → 优先查 ACM

## 7. 输出格式规范 (Report Format)

### 对话框输出（精简）

**执行中**：
```
[Progress] 正在处理 [X/Total]...
```
**仅输出这一行，禁止其他任何描述。**

**完成后**：
```
✅ 核查通过：X 条
⚠️ 字段有误：X 条
❌ 虚假/未找到：X 条

详细报告：verification_report.md
```

### 报告文件 `verification_report.md` 结构（必须严格遵守）

报告分为两部分，按以下顺序输出：

---

#### Part 1：问题条目（⚠️ / ❌ / ❓）

**每条问题文献展示完整信息**，格式如下：

```
### [序号]. [引用Key] — [状态图标] [状态文字]

**问题描述**：[一句话说明哪个字段出错，或为何无法核实]
**原始字段**：`[用户提供的错误字段内容]`
**正确字段**：`[权威来源的正确内容]`
**修正后 BibTeX**：
    [代码块]
**来源链接**：[权威来源 URL]
```

---

#### Part 2：正确条目（✅）

**用一张表格汇总所有核查通过的文献**，格式如下：

| # | 引用Key | 标题（简） | 年份 | 来源 |
|---|---------|-----------|------|------|
| 1 | key1 | Title... | 2024 | CVPR |
| 2 | key2 | Title... | 2023 | IEEE |

表格中**不展示**完整 BibTeX，仅作存档用途。

---

#### 统计摘要（放在报告末尾）

```
✅ 核查通过：X 条
⚠️ 字段有误：X 条
❌ 虚假/未找到：X 条
❓ 无法核实：X 条
共计：X 条
```

## 8. 使用示例

### 示例 1：检查一个 .bib 文件

**用户输入**：
```
帮我检查一下我的毕业论文参考文献 references.bib，我担心有些引用是 ChatGPT 编造的。
```

**Agent 执行流程**：
1. 读取 references.bib 文件
2. 调用 parse_bib.py 解析所有条目
3. 对有 DOI 的条目：并行调用 verify_doi.py 验证
4. 对无 DOI 的条目：并行调用 search 查找
5. 基于片段判定法，给每条引用打标签（✅/⚠️/❌）
6. 生成 verification_report.md
7. 在对话框输出统计摘要

### 示例 2：检查单个引用

**用户输入**：
```
请帮我验证这条引用是否真实存在：
@inproceedings{smith2024novel,
  title={A Novel Approach to Deep Learning},
  author={Smith, John and Doe, Jane},
  booktitle={Proceedings of the IEEE Conference on Computer Vision and Pattern Recognition},
  year={2024}
}
```

**Agent 执行**：
- 提取标题、作者、年份
- 搜索 "A Novel Approach to Deep Learning" John Smith 2024
- 检查搜索结果中是否有匹配的标题和作者
- 返回判定结果

## 9. 常见错误与预防

### 常见错误 1：过度推理
**错误示例**：
```
我注意到第一条引用的标题是 "Deep Learning for..."，而搜索结果中的第一条是 "Deep learning methods for..."，虽然不完全一致，但考虑到大小写和复数形式的差异，而且作者都是 Wang，年份都是 2023，所以我认为这可能是同一篇论文，但为了保险起见，我会标记为需要人工检查...
```

**正确做法**：
```
✅ 已通过：引用 key1
⚠️ 字段有误：引用 key2（年份不匹配）
```

### 常见错误 2：一条条串行处理
**错误示例**：
```
[Turn 1] 用户提供了 10 条引用
[Turn 2] Agent："首先处理第一条引用..." + 调用工具
[Turn 3] Agent："第二条引用..." + 调用工具
...
```

**正确做法**：
```
[Turn 1] 用户提供了 10 条引用
[Turn 2] Agent：一次性并行调用 10 个工具（DOI验证 + 搜索）
```

### 常见错误 3：点开网页获取"更多细节"
**错误示例**：
```
搜索结果有一条标题匹配的，但我需要打开网页确认作者和年份...
[调用 agent-browser 访问网页...]
```

**正确做法**：
```
搜索结果片段已包含标题、年份、第一作者 → 直接基于片段判定
```

## 10. 兼容性

### 依赖要求
- Python 3.8+
- 无需额外 pip 包（仅使用标准库：sys, json, re, urllib）

### 输入格式
- LaTeX .bib 文件（BibTeX 格式）
- 或单个 BibTeX 条目文本

### 验证数据源
- Crossref API（DOI 解析）
- Google Scholar（搜索验证）
